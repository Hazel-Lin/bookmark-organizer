# 我把“整理 Chrome 书签”做成了一个给 Agent 用的开源 CLI

大家好,我是熠萧。

这两天我做了一个很小，但我觉得很有代表性的开源工具：`bookmark-organizer`。

它表面上是在整理 Chrome 书签。
但我真正想解决的，不是“书签太乱”这件事，而是另一件更底层的问题：

**很多 Agent 已经会写代码、会调 API、会查资料，但一到“本地半结构化数据的安全修改”这一步，流程就会变得很脆弱。**

Chrome 书签就是一个很典型的例子。

它不是数据库。
也不是你随便改改就完事的纯文本。
它是一个本地 JSON 文件，背后还可能挂着浏览器运行态、同步机制、历史结构和用户自己的整理习惯。

你如果直接让 Agent 去“帮我整理书签”，大概率会遇到三类问题：

1. 先天不知道该整理哪个文件夹
2. 一旦嵌套层级很多，分类基础就不稳定
3. 最危险的是，改完以后你没法证明“没有丢书签”

所以我这次没有把它做成一个“聪明分类器”，而是先把它做成了一个**可审计的工作流工具**。

它的核心命令只有 4 个：

## 1. scan

先扫描本机书签，列出顶层文件夹、书签数量和子文件夹数量。

这一层的作用不是整理。
而是先回答一个问题：

**到底应该整理哪里？**

这一步对人有用，对 Agent 更有用。
因为 Agent 不能默认知道用户要整理哪个 folder。

## 2. analyze

找到目标文件夹之后，不是直接改结构，而是先把这个文件夹下面所有书签展平。

这一步非常关键。

因为很多人的书签混乱，不是“没有分类”，而是“分类已经套了三四层，但逻辑完全漂了”。

如果不先展平，后面的分类判断会被旧结构污染。

## 3. plan

这是我觉得最重要的一步。

`bookmark-organizer` 不默认直接写回。
它会先基于 taxonomy 生成一份 `plan.json`。

这个 plan 里会明确写出：

- 目标文件夹是谁
- 最终要生成哪些分类
- 每个分类下应该有哪些 bookmark id

也就是说，Agent 先交一份施工图。
而不是先动刀。

这个思路的价值很大。

因为一旦有了 `plan.json`，你就可以：

- 人工 review
- 让另一个 Agent 复核
- 存档
- 反复复用同一套 taxonomy

## 4. apply

只有 plan 确认之后，才真正写回。

而且在写回前：

- 会先备份
- 会校验 plan 覆盖范围
- 默认要求 Chrome 关闭

这意味着它不是“帮你瞎整理书签”。
而是在做一次有回滚、有校验的结构迁移。

## 这件事真正有意思的地方

如果只是为了整理我自己的书签，我完全可以写一个一次性脚本。

但我最后把它做成 CLI，是因为我越来越确认一件事：

**未来很多工具的价值，不是 UI 多漂亮，而是它能不能成为 Agent 可直接调用的稳定接口。**

`bookmark-organizer` 现在做的，其实就是把“书签整理”这件事协议化。

这个协议很简单：

1. 先发现目标
2. 再展平数据
3. 再生成 plan
4. 最后 apply

这套模式不只适用于书签。

它其实也适用于很多本地数据场景，比如：

- 文件夹重组
- 素材库归档
- 本地知识库清理
- 邮件模板重构
- 配置文件批量迁移

你会发现，真正难的从来不是“改”。
而是：

**怎么在改之前，把改动变成一个可以检查、可以追责、可以回滚的计划。**

## 这个项目现在能做什么

目前这个项目已经具备：

- Chrome 书签扫描
- 指定 folder 分析
- taxonomy 驱动分类 plan
- 备份后安全写回
- 机器可消费 JSON 输出

它已经够 Agent 调用了。

当然，它离一个成熟开源产品还有距离。

我下一步更想补的是：

- Edge / Brave 支持
- 重复书签检测
- 死链检查
- 浏览器运行时 API fallback
- TUI 交互确认

## 我这次最大的收获

做完这个工具之后，我对一件事更确定了：

**Agent 时代最值得做的工具，不是“替你完成某个动作”的黑盒，而是“把高风险动作拆成可验证步骤”的白盒。**

这决定了它到底是一次性 demo，还是别人真的敢拿去用的工具。

如果你最近也在做 Agent 工具，或者你手里也有一堆本地脏数据想自动整理，那你可以看看这个方向。

很多人现在在卷模型。
但我越来越觉得，真正的工程杠杆在另一边：

**把任务抽象成一个稳定协议，让任何 Agent 都能接手。**

如果你对这个项目感兴趣，我下一篇可以继续写：

1. 我是怎么把个人书签整理过程抽象成 CLI 协议的
2. 一个 Agent-safe 的 `plan -> apply` 结构到底该怎么设计
3. 为什么“先生成计划，再执行”比“直接自动化”更重要
